# Línea base de performance — 2026-07-14

## Ambiente y metodología

- Equipo local macOS, zona horaria `America/Argentina/Cordoba`.
- Frontend Vite 6.4.3, React 18.3.1, build de producción servido localmente.
- API Express local en `127.0.0.1:4002`, PostgreSQL remoto configurado por el proyecto.
- Perfil medido: Nivel 1 RRHH. Periodo `2026-07`; fecha de asistencia `2026-07-14`.
- Cada endpoint se ejecutó una vez en frío y tres veces en caliente. Los tiempos incluyen red local, API y acceso a la base remota.
- El volumen observado fue pequeño: 25 filas solicitadas; los payloads indican que varios módulos contienen menos registros.
- El navegador integrado no estuvo disponible. Renders React, Web Vitals y tiempos de guardado no se inventaron: quedan pendientes de una sesión visual habilitada.

Comando reproducible después de `npm run build`:

```sh
PERF_EMAIL="usuario" PERF_PASSWORD="clave" npm run performance:baseline
```

## Diagnóstico y decisión de etapa

- Prioridad crítica: Asistencia no presenta mejora suficiente en caliente; requiere inspección de consultas antes de cualquier refactor React.
- Prioridad alta: MediaPipe está incluido en el chunk del fichador aunque la cámara no se abra.
- Prioridad alta: el shell importa `resetDemoData`, que arrastra semillas y servicios mock al grafo productivo.
- Prioridad media: `roleHourOptions` pesa 165 kB y debe quedar limitado a formularios que realmente lo usan.
- La paginación de 25 elementos ya existe en Legajos, Horas, Novedades y Documentación; no se reemplazará.
- Los caches existentes funcionan en Dashboard y listados. La invalidación debe conservarse y ajustarse sólo con evidencia.
- Requests, renders y guardados por interacción quedan pendientes de medición visual. No bloquean los quick wins de bundle demostrados por el build, pero sí bloquean refactors de componentes o mutaciones.

Riesgo pendiente: los cambios locales preexistentes afectan Fichador, Asistencia, Dashboard, documentos y storage. Las siguientes etapas deben integrar cambios mínimos y revisar cada diff para no sobrescribirlos.

## Etapa 3 — Backend prioritario y mocks aislados

Cambios:

- `VITE_DEMO_MODE=false` es el valor documentado y el comportamiento por defecto.
- En modo backend se eliminan una sola vez las claves `losod_demo_*`; sesión, cache geográfica e IndexedDB no se tocan.
- El reset demo sólo aparece con modo demo activo.
- Organigramas dejó de caer silenciosamente a mocks: muestra error claro si fallan ambos endpoints API.
- El dataset demo de organigramas se importa dinámicamente sólo con modo demo explícito.
- Los cálculos del organigrama sobre datos API ya no importan servicios de empleados, puestos o estructura mock.

Medición: carga inicial 324.094 B, 477 B menos que la etapa anterior y 18,2% menos que la línea base. `employeeMockService` desapareció de los chunks principales; el fallback demo quedó en un chunk diferido. Organigramas bajó de 25.108 B a 16.161 B (-35,6%).

Validación: 7 pruebas Vitest correctas y build TypeScript/Vite correcto. Riesgo pendiente: algunos servicios con nombre `MockService` siguen aportando utilidades puras o configuración visual a formularios; no se retirarán sin medir y separar sus responsabilidades.

## Etapa 4 — Listados y requests

Cambios:

- Auditoría ahora consume el contrato paginado existente con 25 filas y muestra paginación real; antes solicitaba hasta 200 y renderizaba el resultado completo.
- Legajos dejó de repetir la request inicial cuando terminaba de cargar el catálogo de empresas. El identificador seleccionado, no el array completo del catálogo, controla la recarga.
- Se conservaron sin cambios las paginaciones ya correctas de Legajos, Horas, Novedades y Documentación.

Medición estructural: Legajos pasa de hasta 4 requests iniciales (`list` duplicada + `summary` + `org-structure`) a 3. Auditoría limita cada render a 25 filas. El payload de referencia para 25 auditorías es 33.534 B, por debajo del presupuesto de 200 kB.

Validación: 7 pruebas correctas y build TypeScript/Vite correcto. Riesgo pendiente: no se aplicó `React.memo` ni se dividieron formularios porque no fue posible obtener un perfil de renders en esta sesión; hacerlo sin profiler violaría el criterio de medición previa.

## Etapa 5 — Guardados puntuales

Cambios:

- Crear, aprobar o rechazar una novedad actualiza la página actual con la entidad devuelta por la API.
- Crear documentación desde el listado inserta el documento devuelto y actualiza el total sin recargar la tabla.
- El servicio documental dejó de ejecutar un GET adicional inmediatamente después del POST; reutiliza los documentos incluidos en la respuesta del empleado.
- Cuando hay búsqueda o una página distinta de la primera, se conserva una revalidación acotada para garantizar el orden y los filtros.

Medición por cantidad de requests: una mutación de novedad pasa de 2 requests (mutación + listado) a 1. Una carga documental desde el listado pasa de 3 (POST + documentos del empleado + listado) a 1. En el detalle de legajo permanece una relectura puntual del empleado, necesaria para refrescar todas sus pestañas.

Validación: 7 pruebas correctas y build TypeScript/Vite correcto. No se modificaron guardados de Fichador, Asistencia, Legajos ni Horas porque sus tiempos visuales no pudieron medirse y contienen trabajo local previo sensible.

## Etapa 6 — Cache e invalidación backend

La medición señaló a Asistencia como el único endpoint persistentemente lento en caliente. Se agregó un cache de 10 segundos segmentado por usuario, rol, fecha y URL completa. Todas las mutaciones de fichadas, jornadas y horas limpian también esta familia; se corrigió específicamente la invalidación faltante al marcar ingreso.

| Asistencia | Antes | Después | Diferencia |
| --- | ---: | ---: | ---: |
| Frío | 3.920,7 ms | 4.372,5 ms | variación de base remota |
| Mediana caliente | 1.979,5 ms | 4,2 ms | -1.975,3 ms (-99,8%) |
| Peor caliente | 2.249,0 ms | 4,5 ms | -2.244,5 ms (-99,8%) |
| Payload | 4.765 B | 4.765 B | sin cambio de contrato |

Validación: typecheck backend correcto y cuatro respuestas HTTP 200 consecutivas. Riesgo pendiente: el primer acceso continúa condicionado por la latencia de PostgreSQL remoto; optimizarlo exige `EXPLAIN ANALYZE` sobre la base representativa y no se agregará un índice especulativo.
